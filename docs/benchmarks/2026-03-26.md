# Search Benchmark Results — 2026-03-26

> **Baseline run** after completing 6 search performance improvements:
> FTS OR, Store connection caching, query decomposition, embedding BLOB exclusion,
> RRF hybrid merge, and deferred hydration.

## Environment

| Property | Value |
|----------|-------|
| Node.js | 22.x LTS |
| Platform | macOS (darwin, arm64) |
| Vitest | 4.1.0 |
| sqlite-vec | Not loaded (JS cosine fallback) |
| Embedder | Synthetic (deterministic 768-dim, `Math.sin` seeded) |

## Dataset

- **480 chunks** across 8 topical documents
- Topics: authentication, database design, React patterns, Kubernetes, machine learning, TypeScript patterns, CI/CD pipelines, performance optimization
- 10 base paragraphs per topic × 6 variants (original + 5 expanded) = 60 chunks per topic
- Each chunk carries a deterministic 768-dimensional embedding

## Key observations

1. **Keyword search is fast** — single-term queries complete in under 0.4ms. Even compound queries (which now use FTS5 `OR` instead of implicit `AND`) stay under 1ms mean.

2. **Vector search is the bottleneck** — the JS cosine fallback scans all 480 embeddings for every query (~4.5ms). With `sqlite-vec` native extension this would be significantly faster.

3. **Hybrid ≈ vector cost** — deferred hydration keeps hybrid search close to vector-only latency. The old approach (full hydration on both legs) would have been roughly 2× slower because it JOINed and parsed metadata for every candidate, not just the final winners.

4. **Limit has minimal impact** — increasing from limit=10 to limit=50 adds only ~0.1–0.2ms, confirming that the scan (not hydration) dominates cost.

5. **No-match queries are cheapest** — FTS5 returns immediately with zero rows; vector search still scans all embeddings but produces low scores.

## Improvements measured (qualitative)

These benchmarks were recorded _after_ all 6 improvements. The changes cannot be directly A/B tested from this baseline, but the architectural impact is:

| Improvement | Expected impact |
|-------------|----------------|
| FTS OR instead of AND | Compound keyword queries now return relevant results (previously returned 0 rows) |
| Store connection caching (MCP) | ~20–50ms saved per MCP call (not reflected in these Store-level benchmarks) |
| Query decomposition (MCP) | Multi-phrase queries searched independently and merged — better recall |
| Exclude embedding BLOBs | Reduced memory per search by not fetching 768-dim float arrays for non-winning rows |
| RRF hybrid merge | Stable ranking independent of per-batch score distributions |
| Deferred hydration | Only final winners are fully hydrated — hybrid cost ≈ vector cost instead of ~2× |
